Key decisions and guidance
- Registry location: src/features/conversation/canvas/registry.tsx.
- Avoid hooks in registry entries; registry should point at components that own their hooks.
- Preserve per-variant state with a wrapper component that calls useSessionOverview and uses key={variant.id}.
- Variant selection should be a Set/array of variant IDs derived from activeDemo.variants (no 'both' | 'a' | 'b').
- Store selection per demo (Record<demoId, string[]>), so switching demos restores prior selections.
- For demos that do not require session data, hide the session picker and render with null/empty data.
- Initial variants should render identical layouts; actual visual differences will be added later.

Canvas registry/data model
- Define a CanvasContext (session data optional) in src/features/conversation/canvas/types.ts.
- CanvasDemoVariant exposes a Component that receives { context }.
- SessionHeader demo variants live in a separate file (sessionHeaderVariants.tsx) and call useSessionOverview internally.

Styling slots to add (semantic, shallow)
- SessionOverview: containerClassName, toggleGridClassName.
- SessionHeader: headerClassName, titleClassName, metaGridClassName, statsRowClassName, actionsClassName.
- Keep defaults identical; only apply optional overrides when provided.

CanvasView behavior
- Demo selector lists canvasDemos entries and shows demo description.
- Variant selector is dynamic from activeDemo.variants and supports multi-select.
- Session picker only appears when activeDemo.requiresSessionData is true.
- Render variants by mapping selected variants and invoking their Component with context.
